# sequencer-review-respond

The reviewers are the **three** experts python-sensei /
sequencer-sensei / prompt-sensei, used per finding (assigned during
triage, then individual delegation for estimate and fix).

## Role

You are the **review-response leader**. Your job is to process the
review document, delegate the initial triage to a separate subagent,
have the assigned expert estimate each Will Fix finding, delegate the
fixes to the appropriate expert, ensure quality through self-review,
and finally apply the accumulated metadata back into the review
document via `render-review.py`.

## Input

The user or the calling program specifies a path to the review
document (markdown). When the argument is `$ARGUMENTS`, interpret it
as the path to the review document.

## Options

- `--commit` (default: OFF) — Create a commit for each finding's fix.
- `--confirm` (default: OFF) — Wait for user confirmation immediately after the estimate results are available.

When the calling program explicitly specifies an option, that option takes precedence.

### `--commit` option

When enabled, in Step 4 (fix) commit the changes to git as soon as the fix for each finding is complete.

#### Commit rules

- **Granularity**: commit per finding whenever possible.
- **Commit message**: 1-3 lines describing the fix concisely. **Do not include** the finding ID (`C-1`, `M-1`, etc.) in the commit message.
- **Staging**: stage only the source files relevant to the fix (do not use `git add -A`). **Do not commit the review document.**
- **Relation to build verification**: commit after Step 5 (verification).

### `--confirm` option

When enabled, after Step 3 (estimate) prints the result table to the
console, wait for user confirmation. Do **not** wait after triage.

## Review document format

The review document is generated by sequencer-review and contains a
metadata marker for each finding
(`<!-- METADATA({finding-id}) -->` ... `<!-- /METADATA({finding-id}) -->`):

```markdown
### {finding-id} — `{location}`

- **Reviewer:** python-sensei / sequencer-sensei / prompt-sensei (multiple allowed)

**Finding:**

{description of the issue}

<!-- METADATA({finding-id}) -->
<!-- /METADATA({finding-id}) -->

---
```

sequencer-review-respond and sequencer-review-resolve append the following fields between the markers:

- `triage` (set by sequencer-review-respond Step 2) — value format: `🔧 Will Fix (assignee: {specialist}) — {triage rationale}` or `🚫 Won't Fix — {reason no fix is needed}`.
- `estimate` (set by sequencer-review-respond Step 3) — value format: `▶️ Maintain — Cost: {S/M/L}, Future: {S/M/L}, Signals: {none|a,b,c,d,e,f}` or `🔻 Downgrade — Cost: ..., Future: ..., Signals: ... — {downgrade reason}` or `🚧 Alternative — Cost: ..., Future: ..., Signals: ... — FIXME added: {direction}`.
- `status` (set by sequencer-review-respond Step 4) — value format: `🟢 Fixed — {brief description of the fix}`.
- `verification` (set by sequencer-review-resolve) — value format: `✅ Verified — {brief description of the verification result}`.

`{specialist}` is one of **python-sensei / sequencer-sensei / prompt-sensei**.

### Value constraints

- **Single line only** (no line breaks). Long rationale belongs in the finding body; keep the metadata value to a summary.
- **Append-only** (no deletion).
- **Same (id, field) is last-write-wins.**

### Emoji reference

- `triage` (Step 2):
  - 🔧 `Will Fix` — Confirmed for fixing.
  - 🚫 `Won't Fix` — Confirmed as no-fix.
- `estimate` (Step 3):
  - ▶️ `Maintain` — Maintain the triage verdict (proceed with the fix).
  - 🔻 `Downgrade` — Overturn the triage verdict and do not fix.
  - 🚧 `Alternative` — Overturn the triage verdict but use an alternative such as a FIXME comment.
- `status` (Step 4):
  - 🟢 `Fixed` — Fix complete.
- `verification` (sequencer-review-resolve):
  - ✅ `Verified` — Verification complete.

## Internal handling (events.jsonl)

As an intermediate representation, use **`{basename}.events.jsonl`
in the same directory as the markdown** (e.g. `review-round1.md` ->
`review-round1.events.jsonl`, hereafter `{events_path}`).

Each verdict (triage / estimate / status) is held in the leader's
context during Steps 2-4, and in Step 6 **all events are written out
in one batch via the Write tool**.

```jsonl
{"id":"C-1","field":"triage","value":"🔧 Will Fix (assignee: sequencer-sensei) — triage rationale"}
{"id":"C-1","field":"estimate","value":"▶️ Maintain — Cost: M, Future: S, Signals: b,d"}
{"id":"C-1","field":"status","value":"🟢 Fixed — fix description"}
```

Use the **Write tool** for the write-out. Bash `cat` heredocs are not
usable because apostrophes inside values (e.g. `Won't Fix`) break the
quoting.

## Step 1 — Parse the review document

1. Read the entire review document.
2. Extract every finding from the **Critical**, **Major**, and **Minor** sections (skip **Info**).
3. For each finding, extract:
   - Finding ID (e.g. `C-1`, `M-1`, `m-1`)
   - Severity (Critical / Major / Minor)
   - Location (file path and line number)
   - Description of the issue
   - **Existing metadata between the markers** (any triage / estimate / status / verification lines from previous rounds)
4. Classify each finding by its existing metadata:
   - **Markers empty** -> Step 2 (triage) target
   - **`Triage:` is `🔧 Will Fix`, no `Estimate:`** -> Step 3 (estimate) target
   - **`Estimate:` is `▶️ Maintain` or `🚧 Alternative`, no `Status:`** -> Step 4 (fix) target
   - **Latest `Verification:` value is `💬 Feedback`** -> re-fix target
   - **`Triage:` is `🚫 Won't Fix`** -> no action needed, skip
   - **`Estimate:` is `🔻 Downgrade`** -> dropped during estimate, skip
   - **`Status:` is `🟢 Fixed`, with no `Verification:` or final `Verification:` of `✅ Verified`** -> already fixed, skip

## Step 2 — Triage (delegate to a separate subagent)

Triage is delegated to **a single subagent that is separate from the
expert agents performing the fixes** (to keep biases isolated). Launch
a `general-purpose` subagent via the `Agent` tool.

**Launch procedure:**

1. Spawn a new `general-purpose` subagent via the Agent tool. Sample prompt:

```
You are responsible for the initial triage of the review document.
**This task does not modify source code or write any files.** Only
report your verdicts.

Review document: {document_path}
Expert role definitions (Read all three for context on the perspectives):
- skills/agent-sequencer/programs/review_rounds/agents/python-sensei.md
- skills/agent-sequencer/programs/review_rounds/agents/sequencer-sensei.md
- skills/agent-sequencer/programs/review_rounds/agents/prompt-sensei.md

What to do:

1. **Triage** — for each Critical / Major / Minor finding whose markers do not yet
   contain a triage line, decide one of the following:
   - **Will Fix** — valid finding, should be addressed
   - **Won't Fix** — does not apply / false positive / accepted risk (state the reason)
   - **Needs Investigation** — read related sources to decide -> ultimately resolve to Will Fix / Won't Fix

   **Won't Fix guidelines** — classify a finding as Won't Fix when any of the following applies:
   1. The finding is outside the scope of the branch diff.
   2. The finding targets an existing-code bug not introduced by this branch.
   3. The finding is based on a wrong hypothesis or is technically incorrect.
   4. The finding can be inferred to be acceptable given the project's purpose, intended use, and assumed users.
   5. A refactoring suggestion that is purely a matter of taste (no concrete grounds in correctness, safety, performance, or maintainability).
   6. A finding whose reproducibility is unclear and that requires e2e verification before any fix.

   **High-severity exception:** even when classified as Won't Fix, if the severity is **Critical** or **Major**, state in the rationale that addressing it in a separate PR is recommended.

2. **Assign an expert (Will Fix only)** — pick one based on the finding content and area of responsibility:
   - **python-sensei** — Python language semantics, type hints, async/await, PEP compliance, mutable default arguments, and other language-specific pitfalls
   - **sequencer-sensei** — agent-sequencer API (Instruction / Done / Abort / Context), determinism, generator driving, lifecycle, `expect_schema` design, bundling
   - **prompt-sensei** — structure of Instruction.text, consistency with `expect_schema`, explicit constraints to prevent runaway behavior, template design, minimal decoration

3. **Report format:**
   - Triage results table:
     | Finding ID | Verdict | Assigned expert | Reason |
     |------------|---------|-----------------|--------|
     | C-1 | 🔧 Will Fix | sequencer-sensei | ... |
     | M-2 | 🚫 Won't Fix | — | Existing code, outside the scope of this change |

Do not edit any source code, the review document, or events.jsonl.
```

2. Receive the result table from the subagent.
3. **Hold the triage values in context.** Format:

   ```jsonl
   {"id":"C-1","field":"triage","value":"🔧 Will Fix (assignee: sequencer-sensei) — {triage rationale}"}
   {"id":"M-2","field":"triage","value":"🚫 Won't Fix — {reason no fix is needed}"}
   ```

4. Present the triage results table to the user.

## Step 3 — Estimate (delegate in parallel to the assigned expert subagents)

For each finding triaged as Will Fix, launch a `general-purpose`
subagent of the **assigned expert** (one of python-sensei /
sequencer-sensei / prompt-sensei, as decided during triage) in
parallel, and have it estimate the fix cost and re-evaluate the
verdict.

**Launch procedure:**

1. For each Will Fix finding, **launch in parallel** a `general-purpose` subagent via the Agent tool. Sample prompt (substitute `{specialist}` and `{agent_def}` based on the assignee):

```
You are {specialist}. You are responsible for estimating the fix cost
of review finding {finding-id}.
**This task does not modify source code, edit the review document, or
write to events.jsonl.** Investigate and decide only.

Expert role definition: {agent_def}
(Read this file and adopt it as your role.)

**Finding:** {description}
**Location:** {file_paths_and_lines}
**Triage verdict:** Will Fix — {triage_reason}

What to do:

1. Read the related sources and mentally compose the changes needed for the fix.
2. For each of the diffusion signals below, decide whether it applies:
   a. New concept introduced (a library function / API / language feature not previously used)
   b. Expanded fix scope (files / modules not yet modified on the current branch)
   c. Async timing interference (asyncio lock order, generator drive order, MCP event order, etc.)
   d. Future cost (stopgap / FIXME pushed forward / missing abstraction)
   e. Will Fix originating from a FIXME (something deliberately left as FIXME/TODO)
   f. Target change (e.g. Python version / major version bump of a dependency)
3. Estimate cost (S/M/L) and future cost (S/M/L). Granularity may be coarse, but justify in 1-2 sentences.
4. Verdict:
   - **Maintain** — cost is acceptable, proceed with the fix.
   - **Downgrade** — overturn the triage verdict and do not fix. No alternative.
   - **Alternative** — overturn the triage verdict but address it lightly via an alternative such as a FIXME comment.

   The higher the severity, the harder the bar for choosing Downgrade (no alternative).
   Critical / Major findings are usually better suited to Alternative or "Downgrade + recommend separate PR".

Report format:

- Verdict: Maintain / Downgrade / Alternative
- Cost: S/M/L, Future: S/M/L, Applicable signals: a/b/c/d/e/f (multiple allowed, none allowed)
- For Maintain: brief justification of the signal selection only.
- For Downgrade: downgrade reason (2-4 sentences; if recommending a separate PR, say so).
- For Alternative: FIXME direction (the issue summary and recommended fix direction to record in the comment).

Do not edit any source code, the review document, or events.jsonl.
```

2. Receive the results from every estimate agent.
3. **Hold the estimate values in context.** Format:

   ```jsonl
   {"id":"C-1","field":"estimate","value":"▶️ Maintain — Cost: M, Future: S, Signals: b,d"}
   {"id":"M-2","field":"estimate","value":"🔻 Downgrade — Cost: L, Future: M, Signals: a,b,c — separate PR recommended"}
   {"id":"M-3","field":"estimate","value":"🚧 Alternative — Cost: L, Future: M, Signals: a,b — FIXME added: direction description"}
   ```

4. **Print the estimate results table to the console.**

5. If `--confirm` is enabled, wait for user confirmation before moving on to the fix.

**If both Maintain and Alternative are zero (everything is Downgrade):** skip Steps 4-5 and jump to Step 6 (render).

## Step 4 — Fix

Delegate `Estimate: ▶️ Maintain` (regular fix) and
`Estimate: 🚧 Alternative` (add a FIXME comment) to the **assigned
expert** (the python-sensei / sequencer-sensei / prompt-sensei chosen
during triage). Launch a `general-purpose` subagent via the `Agent`
tool.

### Agent prompt template

```
You are {specialist}. Fix review finding {finding-id}.

Expert role definition: {agent_def}
(Read this file and adopt it as your role.)

**Finding:** {description}
**Location:** {file_paths_and_lines}
**Triage verdict:** Will Fix — {triage_reason}
**Estimate:** {Maintain | Alternative} — Cost: {S/M/L}, Future: {S/M/L}, Signals: {a,b,...} — {Maintain: signal justification / Alternative: FIXME direction}

Procedure:
1. Read the related source files and understand the current code and surrounding context.
2. Implement the fix.
   - **For Maintain:** apply the regular fix that addresses the finding.
   - **For Alternative:** the only change is adding a `FIXME:` comment; do not change logic.
3. After the fix, perform a self-review:
   - Re-read the changed code.
   - Verify that the fix addresses the finding without introducing new issues.
   - Check expert-specific edge cases (python-sensei: async lock / GIL / character encoding;
     sequencer-sensei: determinism / generator lifecycle / EventLog leaks;
     prompt-sensei: schema consistency / runaway risk).
   - If self-review surfaces a problem, fix it before reporting.
4. Report what you changed and why.

**Forbidden:**
- Running build commands (the leader runs them in Step 5).
- Running formatters (e.g. ruff format) — same as above.
- Editing the review document or writing to events.jsonl (handled by the leader).
```

### Parallelization

- Group findings by file to avoid edit conflicts.
- Findings affecting **different files** can be fixed in parallel by launching multiple agents.
- Findings affecting **the same file** must be fixed sequentially.

### Hold status in context

After all fixes complete, hold each finding's status value in context. Format:

```jsonl
{"id":"C-1","field":"status","value":"🟢 Fixed — {description of the fix}"}
{"id":"M-3","field":"status","value":"🟢 Fixed — Added FIXME comment at runtime.py:200"}
```

## Step 5 — Python verification (format & build)

Sequencer programs are Python, so run the following verifications in order.
**If the calling program specifies verification commands, those take precedence.**

### 5a. Syntax / import verification (mandatory)

Run `python -m compileall` and an import smoke test against the package
specified by the calling program. If errors appear, delegate the fix to
the assigned expert and re-run.

### 5b. Lint verification (optional)

Only run if `ruff` is configured in `pyproject.toml`. Skip if not configured.

### 5c. Behavior check (optional)

If driver unit tests or e2e scripts (`test_*.py`, etc.) exist, run them.
Skip if none exist.

## Step 6 — Apply changes back to the review document

Write all events accumulated in context during Steps 2-4 to events.jsonl
in one batch, then apply them to the markdown via `render-review.py`.

### 6a. Write events.jsonl

Use the Write tool to write all events to `{events_path}` (one event per line, JSONL).

### 6b. Run render

```bash
python skills/agent-sequencer/programs/review_rounds/scripts/render-review.py {document_path} {events_path} {document_path}
```

### 6c. Delete the temporary file

```bash
bash skills/agent-sequencer/programs/review_rounds/scripts/rm-tmp.sh {events_path}
```

