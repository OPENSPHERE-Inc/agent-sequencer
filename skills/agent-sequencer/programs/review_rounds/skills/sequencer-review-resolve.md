# sequencer-review-resolve

You are the **review verifier**. Your job is to re-read the updated
review document, verify each finding's resolution status against the
actual source code, and write the verification result back as
`verification` metadata.

## Input

The user specifies a path to the review document (markdown). When the
argument is `$ARGUMENTS`, interpret it as the path to the review
document.

## Review document format

The review document is generated by sequencer-review and contains a
metadata marker for each finding. It is assumed that
sequencer-review-respond has already added the `triage` / `estimate` /
`status` fields:

```markdown
### {finding-id} — `{location}`

- **Reviewer:** {reviewer name}

**Finding:**

{description}

<!-- METADATA({finding-id}) -->
- **Triage:** 🔧 Will Fix (assignee: cpp-sensei) — valid finding
- **Estimate:** ▶️ Maintain — Cost: M, Future: S, Signals: b,d
- **Status:** 🟢 Fixed — Added a null check
<!-- /METADATA({finding-id}) -->

---
```

The current state of each finding is decided from **the fields present
between the markers**:

- None — Not yet triaged.
- Only `Triage: 🔧 Will Fix` — Triaged, estimate pending.
- `Triage: 🚫 Won't Fix` — Confirmed as no-fix.
- `Estimate: ▶️ Maintain`, no `Status:` — Estimated, fix pending.
- `Estimate: 🔻 Downgrade` — Dropped during estimate.
- `Estimate: 🚧 Alternative`, no `Status:` — Estimate chose alternative; FIXME not yet added.
- `Status: 🟢 Fixed` — Fix complete (Maintain fix or Alternative FIXME added).

sequencer-review-resolve appends the `verification` field in the following form:

- `✅ Verified — {brief description of the verification result}` — Resolved.
- `💬 Feedback — {what is missing and what is needed for full resolution}` — Feedback required.

For Unresolved findings, do not write `verification` (those are not at the verification stage yet).

### Value constraints

- **Single line only** (no line breaks). Long descriptions belong in other sections (such as the feedback details); keep the metadata value to a summary.
- **Append-only** (no deletion).

## Internal handling (events.jsonl)

As an intermediate representation, use **`{basename}.events.jsonl`
in the same directory as the markdown** (e.g. `review-round1.md` ->
`review-round1.events.jsonl`, hereafter `{events_path}`).

The verification verdict for each finding is held in the leader's
context during Step 2, and in Step 4 **all events are written out in
one batch via the Write tool**.

```jsonl
{"id":"C-1","field":"verification","value":"✅ Verified — Null check fix is correct"}
{"id":"M-1","field":"verification","value":"💬 Feedback — The else branch at line 85 is missing LOG_ERROR"}
```

Use the **Write tool** for the write-out. Bash `cat` heredocs are not
usable because apostrophes inside values (e.g. `Won't`) break the
quoting.

## Step 1 — Re-read and parse

1. Read the entire review document.
2. Extract every finding from the **Critical**, **Major**, and **Minor** sections.
3. For each finding, extract:
   - Finding ID, severity, location (file and line), description
   - The metadata between the markers (presence and values of triage / estimate / status)

## Step 2 — Verify each finding

### `Status: 🟢 Fixed` present

1. Read the referenced file and lines.
2. Confirm that the recorded fix actually exists in the code.
   - **When Estimate is `▶️ Maintain`:** confirm that the regular fix (including logic changes) is in place and fully addresses the original issue.
   - **When Estimate is `🚧 Alternative`:** confirm that an appropriate `FIXME:` / `TODO:` comment has been added at the location. Check that the comment text roughly matches the FIXME direction recorded in Estimate, and that it carries enough information for the future fix. Logic changes are not expected.
3. Confirm that the fix has not introduced any new problems (regressions, new bugs, style violations, thread-safety issues, resource leaks, etc.).
4. Verdict:
   - **Resolved** — the fix is correct, complete, and introduces no new issues.
   - **Feedback** — the fix is missing, incomplete, or introduces new issues. Describe what remains.

### `Triage: 🚫 Won't Fix` present

1. Read the referenced file to understand the current state.
2. Evaluate whether the rationale for "no fix" still holds against the current code.
3. Verdict:
   - **Resolved** — the rationale is sound and the verdict is appropriate.
   - **Feedback** — the rationale has flaws, or the situation has changed. Explain why.

### `Estimate: 🔻 Downgrade` present

1. Read the referenced file to understand the current state.
2. Evaluate whether the downgrade rationale (diffusion signals, Cost, Future, verdict reason) still holds against the current code.
3. If a separate-PR recommendation is appended to the value, confirm that decision is sound (Critical / Major findings without a separate-PR recommendation deserve especially careful scrutiny).
4. Verdict:
   - **Resolved** — the downgrade rationale is sound.
   - **Feedback** — the downgrade rationale has flaws, or the situation has changed. Explain why.

### `Estimate: 🚧 Alternative` present, no `Status`

- Report as **Unresolved** — the estimate scheduled an alternative response (FIXME), but the FIXME comment has not been added yet.

### `Estimate: ▶️ Maintain` present, no `Status`

- Report as **Unresolved** — the estimate decided Maintain, but the fix has not been completed.

### Only `Triage: 🔧 Will Fix`

- Report as **Unresolved** — triage is done but the estimate is not.

### No metadata between the markers at all

- Report as **Unresolved** — has not been triaged yet.

## Step 3 — Verification report

Print the verification report in the following format **to the
console** (do not write it to any file):

```
# Review verification report

**Review document:** {path}
**Verified on:** YYYY-MM-DD

## Results

### Resolved

| # | Severity | Final field | Verdict |
|---|----------|-------------|---------|
| C-1 | Critical | Status: 🟢 Fixed | Fix is correct and complete. |
| M-2 | Major | Triage: 🚫 Won't Fix | Rationale is sound. |

### Feedback required

| # | Severity | Final field | Issue |
|---|----------|-------------|-------|
| M-1 | Major | Status: 🟢 Fixed | The fix addresses the null check, but the `else` branch at line 85 returns without logging. |

### Unresolved

| # | Severity | Final field | Note |
|---|----------|-------------|------|
| m-3 | Minor | (none) | No verdict line recorded (not yet triaged). |
| m-5 | Minor | Estimate: ▶️ Maintain | Estimated but the fix has not been completed. |

## Summary

- **Findings verified:** N
- **Resolved:** N
- **Feedback required:** N
- **Unresolved:** N
```

### Feedback details

For each finding requiring feedback, append a detailed description after the summary table:

```
---

### M-1 — Feedback

**Original finding:** {brief description}
**Final field:** Status: 🟢 Fixed
**Actual state:** {what was observed in the code}
**Issue:** {what is incorrect or incomplete}
**Suggestion:** {what to do for full resolution}

---
```

## Step 4 — Apply changes back to the review document

Write all verification events accumulated in context during Step 2 to
events.jsonl in one batch, then apply them to the markdown via
`render-review.py`.

### 4a. Write events.jsonl

Use the Write tool to write all events to `{events_path}` (one event
per line, JSONL). See the format in the "Internal handling" section.
**Do not write events for Unresolved findings** (they are not at the
verification stage yet).

### 4b. Run render

```bash
python skills/agent-sequencer/programs/review_rounds/scripts/render-review.py {document_path} {events_path} {document_path}
```

### 4c. Delete the temporary file

```bash
skills/agent-sequencer/programs/review_rounds/scripts/rm-tmp.sh {events_path}
```
